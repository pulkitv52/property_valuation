## 1. Purpose of This Report
This report provides a single end-to-end view of the current PoC pipeline up to Phase 7. It is intended to help explain:
- what raw datasets were used from the `data/` folder
- what cleaning and preprocessing were performed
- how GIS layers were processed and merged
- what feature engineering has been completed
- what modeling and evaluation have been completed
- what is still missing and where the PoC can be improved

This report reflects the actual outputs generated by the current codebase and summary artifacts, including the latest EDA and tuned XGBoost evaluation.

### Phase 4: Data Merge
Code modules used:
- `src/data_merge.py`
- `src/run_phase_4_data_merge.py`

What was done:
- created normalized merge keys on both transaction and property data:
  - `district_name_norm`
  - `ps_code_norm`
  - `mouza_code_norm`
  - `plot_no_norm`
- aggregated duplicate property keys before merge to avoid row explosion
- left-merged property GIS attributes and geometry onto cleaned transactions

Merge strategy:
- transaction rows were preserved as the main grain
- property layer was dissolved/aggregated to one record per merge key
- geometry and selected parcel/admin attributes were carried into the merged layer where keys matched

Phase 4 results:
- transaction input rows: `273,587`
- property input rows: `111,497`
- unique property merge keys: `93,775`
- duplicate property merge-key rows: `25,532`
- matched transaction rows: `207,442`
- unmatched transaction rows: `66,145`
- match coverage: `75.82%`

What this means:
- about three-fourths of cleaned transactions could be linked to property GIS geometry
- the remaining one-fourth do not have parcel geometry in the merged base dataset
- those unmatched rows still remain in the modeling dataset, but some spatial features are missing for them

### Phase 6: Model Training
Code modules used:
- `src/model_training.py`
- `src/run_phase_6_model_training.py`

Target definition:
- model target is always `value_per_area`
- training transform: `log1p(value_per_area)`

Modeling setup:
- train/test split: `80/20`
- random state: `42`
- comparison performed across candidate models
- current winner selected using: `MAPE`, then `RMSE`, then `R²`

Leakage-safe locality encoding added:
- target-encoding is learned only from the training fold, not from the full dataset
- locality columns used for target encoding:
  - `property_district_Name`
  - `PS_Name`
  - `Mouza_Name`
  - `Zone_no`
  - `Road_Name`
  - `Road_code`
  - `GP`

Candidate models compared in the latest run:
1. `RandomForestRegressor`
2. `ExtraTreesRegressor`
3. larger `ExtraTreesRegressor`
4. `HistGradientBoostingRegressor`
5. `XGBRegressor`
6. `LGBMRegressor`

Focused XGBoost tuning was then performed around the first winning XGBoost configuration, producing three tuned variants: `xgboost_deep__full`, `xgboost_regularized__full`, and `xgboost_wide__full`.

Winning model in the latest completed run:
- `xgboost_deep__full`

Winning training summary:
- input rows: `273,587`
- training pool rows used: `273,587`
- train rows: `218,869`
- test rows: `54,718`
- sampled training rows fitted: `80,000`
- model: `XGBRegressor`
- preprocessing: one-hot encoding + target encoding + numeric imputation

## 8. What Is Missing or Can Improve the Model Further
The following are the most important improvement opportunities.

### 8.1 Better locality and neighborhood signals
Potential improvements:
- train-only locality aggregates at more levels
- broader neighborhood encodings such as municipal, ward, block, and GP hierarchies where coverage allows
- local market trend features by time and geography
- locality medians, percentiles, and volatility measures, not just target means

Why this matters:
- property valuation is strongly location-driven
- better neighborhood context can materially improve model fit

### 8.2 Better handling of unmatched spatial rows
Current issue:
- `66,145` rows lack matched parcel geometry

Possible improvements:
- improve merge strategy with alternate keys
- attempt fuzzy/secondary matching on road, mouza, bata, or alternate parcel fields
- build backup locality-only spatial proxies for unmatched rows

Why this matters:
- more matched geometry would increase spatial signal coverage and reduce feature missingness

### 8.3 More road-network features
Current road features are useful, but can be expanded.
Potential additions:
- road density around parcel
- connectivity or intersection density
- road-class composition within local buffers
- road-length within 250m / 500m / 1km

Why this matters:
- road access and network context are important valuation drivers, especially for land and mixed peri-urban regions

### 8.4 More facility-context features
Potential additions:
- richer use of facility attributes such as `Survey_Use`, `Office_Nam`, `Police_Sta`, `Block__ULB`, `GP_Ward`
- facility density by administrative type and facility function
- distance to specific classes such as health, education, markets, transport nodes

### 8.5 Outlier and segment-specific modeling
Potential improvements:
- stronger outlier handling for `value_per_area`
- separate models for `Flat_or_Land`
- separate urban vs rural models
- segment-specific models by transaction type or geography

Why this matters:
- a single global model may underfit very different property submarkets
- the EDA already showed that `Land` and `Flat` have dramatically different target distributions, making segmentation a strong next experiment

### 8.6 Stronger model families
Current best official model is a tuned `XGBRegressor` (`xgboost_deep__full`).
Potential next candidates:
- CatBoost
- segment-specific XGBoost models
- feature-importance-guided pruning and tuning

Why this matters:
- boosted tree methods often outperform bagged trees on heterogeneous tabular valuation problems

### 8.7 Explainability
Still missing from the plan:
- feature importance report
- SHAP or other local explanation method
- high/low valuation factor breakdown

This is important for stakeholder trust in the PoC.

## 9. Recommended Next Technical Priorities Before Phase 8
If the goal is to maximize model quality before moving into zone clustering, the highest-value next steps are:

1. Improve parcel-match coverage
- reduce the `66,145` unmatched rows through alternate parcel-key and locality-assisted matching strategies

2. Add stronger locality aggregates
- build train-only locality median, percentile, and volatility features by mouza, road, GP, zone, and PS

3. Benchmark segmented models
- compare the current global tuned XGBoost against:
  - separate `Land` and `Flat` models
  - optionally separate `Urban` and `Rural` models

4. Expand road-network context
- add density, connectivity, and nearby major-road features instead of only nearest-road descriptors

5. Add explainability
- implement SHAP-based explanations for the current tuned XGBoost model before stakeholder-facing demo work
